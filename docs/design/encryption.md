## 0. 一句话讲清（给非技术读者）—— 已按红队诚实改写

- 你的账本被一把**随机钥匙**锁住；这把钥匙再被**电脑的安全芯片(TPM)**收着，要**在这台电脑上 + 输对密码**才放出来。
- **把数据库文件拷到别的电脑：解不开**（钥匙绑这台芯片、拿不出来，没有"光靠密码算出钥匙"的路）。这一档是真的、比纯软件强。
- **但有个诚实的差距**：iPhone 的"错 10 次抹掉"是**芯片自己**在硬件里数、自己抹，系统都改不动；**电脑的安全芯片没有这种"给我这把钥匙单独数到 5 就毁掉"的能力**——它的防猜测是全局的、由系统策略管（通常是"锁一阵子"不是"毁钥匙"）。
- 所以"错 5 次销毁"在电脑上有两种做法：**硬路**＝用芯片里的"单调计数器+策略"让芯片自己超 5 次就拒绝放钥匙（最接近 iPhone，但**难、得先验证做不做得出来**）；**软路**＝我们软件来数、到 5 删钥匙（简单，但对手拔电/拷文件能躲过那一次销毁）。**本期先做技术验证(Spike)看硬路行不行，行就用硬路。**
- **代价（你已确认接受）**：本地**没有后门**。忘密 / 清芯片 / 换坏电脑 = 数据**彻底找不回**。这是这套强度的代价，跟 iPhone 一样。所以**一定要自己留备份**（将来用端到端云同步当正式找回，但见 §8 的单机限制）。

## 1. 目标与威胁模型（含全部诚实边界）

**威胁模型 ＝ A「设备级隐私」+ 尽力而为的硬清空**：防捡到/借用/共用这台电脑的人翻看你的账。

**这套能给的（真）**：
- **拷文件无法离线爆破**：DEK 是随机数、被安全芯片绑定封装、拿不出芯片；芯片对解封的密码尝试有（全局）防猜测限速。**这是比纯软件强的核心一档。**

**这套给不了 / 不防（必须 UI/文档讲清，不夸大）**：
- **电脑 TPM ≠ iPhone 安全岛**：「错 N 次销毁」在 Windows 上是**软件触发的尽力而为**（app 数到 N 再删钥匙），**对手在第 N 次前拔电 / 杀进程 / 把封装文件拷走离线保留，就能阻止那次销毁**。硬件自治的"数错 N 次毁钥匙"要靠 §5 的"硬路"(NV 单调计数器+策略)才逼近，且待 Spike 证实。
- **TPM 防猜测是全局的**：和 BitLocker / Windows Hello 共用一个计数器，会互相干扰；阈值/恢复时间是系统策略、app 改不了。密码输错可能连带触发系统级锁定。
- **物理接触者可"清 TPM"删库**：进 UEFI 或管理员可 Clear TPM → 封装钥匙没了 = 数据永久不可解，**不需要密码、绕过一切二次确认**。这是物理对手最省事的破坏路径（破坏性、非窃取）。
- **总线嗅探 / 故障注入**：独立 TPM 芯片(dTPM)的 CPU↔芯片总线若未加密会话，物理接触者可嗅探截获解封数据（BitLocker 已有实证）；固件 TPM(fTPM) 有 faulTPM(电压注入)/TPM-Fail(时序)等公开攻击。→ §10 必须启用**加密+HMAC 的 TPM 会话**，否则"拿到芯片也很难"不成立。
- **解锁运行期**：解出的明文在内存，OS 可能换页到 swap / 休眠写 `hiberfil.sys`；解锁后的进程内存含 DEK。缓解靠 OS 整盘加密(BitLocker)。
- **设密码前的历史明文 / 迁移删除残留**：SSD 的 TRIM/wear-leveling/CoW 使"覆盖删除"不保证物理擦除。
- **运行时内存取证 / 恶意软件 / 国家级**：能力之外。
- **诚实上限**：综合以上，定位是「**对拿到这台电脑的普通人很难**」，**不是**「绝对不可恢复」。

诚实定位一句话：**「设了密码后，拷文件到别处解不开（钥匙锁在这台芯片里）；但'错 5 次彻底销毁'在电脑上做不到 iPhone 那种 100% 硬件自主——我们会先验证能不能用芯片做到接近，做不到就如实告诉你它只是'尽力销毁'。无论哪种，忘了密码就是彻底没了。」**

## 2. 不在范围 / opt-in / 三态状态（红队：防"以为强其实弱"）

- 浏览器演示版：无磁盘文件，不涉及。
- **加密 opt-in**：默认明文＝沿用现状，不强迫随手记账的个体户加门。
- **状态行三态（必做）**：设置页「安全」卡常驻显示，**区分强/弱/无**：
  - `未加密（明文）`
  - `已加密 · 安全芯片保护（强）`
  - `已加密 · 纯软件（较弱：本机无安全芯片，拷文件理论上可离线尝试破解、且不支持硬销毁）`——区别色 + 一句解释。
- **绝不静默降级**：在有芯片的机器设密码(强版)、换到无芯片老机恢复 → 必须模态告知已降级到弱版，不能照旧显示"已加密"。
- core / 业务逻辑 / SQL：**不动**。

## 3. 密钥架构（随机 DEK + 安全芯片绑定封装）

- **DEK（数据钥匙）**：随机 256-bit，**真正加密数据库**（SQLCipher 的 key）。不是从密码算的——这是"拷文件无法离线爆破"的根。
- **安全芯片封装**：用 **NCrypt + Microsoft 平台加密提供程序(PCP)** 在 TPM 里建一把**不可导出**的封装钥匙；密码作为这把钥匙的**使用授权(PIN/usage-auth)**。解封 DEK 必须 ① 在本机芯片上 ② 输对密码。
  - **诚实边界（红队）**：PCP 能给"非导出 key + 密码授权 + 全局 DA 限速"；但**拿不到/改不了 TPM 的精确失败计数与阈值**（那是系统级策略）。所以"错 5 次"的**计数与销毁判定要 app 自管**（§5），别假设能读芯片硬件计数。
- **TPM 会话加密**：所有与 TPM 的交互用 salted/bound 的加密+HMAC 会话，挡总线嗅探（§1）。
- **改密码 = 改这把封装钥匙的授权口令**（走 §6 的 TPM 对象切换原子协议，**不动数据库密文**，秒级）。

> 与 v3 比：v3 含糊地说"芯片硬件里数次数"；v4 改为"芯片只负责绑定+授权+全局限速，'错 N 次'计数与销毁由 app 自管（硬路再把计数交给芯片策略，见 §5）"。SQLCipher 自管页级加密的 nonce/完整性，v2 路线 B 的 nonce/写放大几条不适用。

## 4. 本地库加密落地：路线 A · SQLCipher（Rust）—— 含被低估的真实成本

选 **路线 A（SQLCipher in Rust）**：芯片只能 Rust 对接 → 钥匙+加密+将来云端全在 Rust 最自洽；SQLCipher 透明加密、增量写、崩溃安全。

**红队修正的真实工作量（不是"薄桥可平替"那么轻）**：
- **构建链是另一个量级**：`rusqlite` 开 `bundled-sqlcipher-vendored-openssl` → 编 SQLCipher C 源 + vendored OpenSSL（libcrypto 做 AES/HMAC），Windows MSVC 下需 **Perl（可能 nasm）工具链**；当前 Cargo.toml 是纯 Rust 无 C 依赖，差距大。**Spike 第一关就是它（最可能因 OpenSSL/Perl 翻车）。**
- **不能与 tauri-plugin-sql 并存**：同一 cdylib 内两份 sqlite C 符号冲突 → 必须**整体替换** tauri-plugin-sql（去插件、Cargo 改、lib.rs 注册自定义 `#[tauri::command]`、`capabilities/default.json` 把 `sql:*` 权限换成新命令）。
- **自写桥要复刻 sqlx 的形状**（不只是 load/select/execute 三件套）：
  - 占位符 **`$1..$N` → rusqlite 的 `?N`** 翻译层（现有 SQL 通篇 sqlx 风格）。
  - 返回**用 column_name 显式映射成 {列名:值} JSON**（让 schema.ts 的 `toX` 零改，含 INTEGER/REAL/NULL 对齐）。
  - **单连接 `Mutex<Connection>`**——顺带修掉现有"连接池下放弃事务"老债（事务红利在此自然到手，但仍需把多写方法包进 BEGIN/COMMIT）。
  - **`PRAGMA key` 必须是开库第一条**，再 `journal_mode/foreign_keys/busy_timeout`、再读 `user_version` 迁移；顺序错报 "file is not a database"。WAL 下 `-wal/-shm` sidecar 也是密文。
- **正面已核实**：现有调用确实只 `select/execute` 且从不读 execute 返回值（每次写后 `getX` 重查）→ "写接口干净"这半句成立。
- 路线 C（解密到临时明文文件）仍**否决**。

## 5. 硬清空（错 N 次销毁）—— 两条路，本期先 Spike 硬路

**用户拍板：先 Spike 硬路，做得出用硬路、做不出退软路并诚实标注。**

### 硬路（最接近 iPhone，待 Spike 证实可行）
- 用 TPM 的 **NV 单调计数器**（`TPM2_NV_Increment`，只增不可回滚）记失败次数；把"**计数 < N**"写进封装对象的 **authPolicy**（`PolicyNV`）——这样**不靠 app 主动删钥匙**：计数一旦超 N，**TPM 自己拒绝解封 DEK**，且计数不可回滚、不可被快照回退。这才逼近"iPhone 式确定性、拔电也躲不过"。
- **Spike 必验**：Windows 上能否（经 tss-esapi/TBS）建 NV 计数器 + 设 PolicyNV + 用普通用户态权限增计数；阈值/owner-auth 等约束。**有真实翻车风险**，故先验。

### 软路（退路，简单但弱）
- app 自管失败计数（**原子写、且与"成功解锁"绑定归零**），到 N 次 `NCryptDeleteKey` 删封装钥匙。
- **诚实局限**：app 触发 → 对手第 N 次前拔电/杀进程/拷封装文件可阻止那次销毁；真护城河仍是"强密码 + 拷文件解不开"。

### 两路共用的纪律（红队）
- **三类失败分流（防误删，关键）**：① **密码错** → 计入销毁计数；② **文件/封装损坏**（坏块等）→ 独立"数据可能损坏"分支；③ **芯片态异常**（TPM 被占用/句柄失败/睡眠唤醒上下文失效/BIOS 变更后不可用）→ **独立"安全芯片暂时不可用，请重启重试"分支，绝不计数**。解锁前先 ping TPM 健康，不健康直接走 ③、不让输密码。**销毁计数只由 app"仅密码错才+1"判定，不直接读会被污染的 TPM 全局 DA。**
- **开销毁＝强闸门（本期无云兜底，红队定为灾难性，必做）**：(1) **开启销毁前必须先成功导出一次备份**（强制，不是建议）；(2) **销毁默认带"后悔药"**——先把密文库移到**带时间戳的隔离区**再删（仍无 DEK 不可解、但给一次物理找回机会），本期默认必带、不可在不懂后果时关掉；(3) 文案量化后果：「将**永久且不可恢复**删除你**全部账本**，本机无任何找回途径」；(4) 对"连续快速失败"做节流，降低熊孩子乱按一口气达阈。
- **销毁终态标记**：销毁时写一个持久 sentinel；下次启动**优先识别**，进明确的"数据已按你的安全设置销毁、无法恢复"终态屏，与"数据可能损坏(可重试)""芯片暂不可用(请重启)"三屏彻底分开（否则用户对着乱码反复重启徒劳）。
- **反噬警告**：开了销毁，等于把"删库"递给能物理碰电脑的人（故意输错 N 次；甚至直接 Clear TPM 更省事）——UI 诚实点出，这是"可选 + 默认关 + 强确认"的理由。

## 6. 解锁 UX + 设置

- **解锁屏**（仅已加密）：密码框；失败按 §5 三类分屏；开了销毁显"再错 N 次将永久销毁，本机无法找回 + 当前已错次数"。芯片不可用走专门屏。
- **设置「安全」卡**：三态状态行(§2) + 设/改/移除密码 + 「错N次销毁」开关(默认关、开前强制备份、强警告) + **上次备份：N 天前**(超期软提醒) + 可选自动锁。
- **改密码 / 移除密码 = TPM 封装对象切换，需独立两阶段原子协议**（红队：§9 的文件迁移协议救不了 TPM 对象切换）：先用新密码封出 `heng.dek.tpm.new` + fsync + 验证可解 → 原子顶替 → 最后才 evict 旧对象；**启动自愈要能认旧/新两个封装文件、用任一可解者恢复**，杜绝"改密码断电后新旧都解不出=静默锁死"。
- **移除密码的后果文案**（红队）：必须讲清"移除＝数据库变回明文，任何拿到文件的人都能直接打开，硬销毁也一并失效"；按钮文案改后果导向"移除密码并解密为明文"。
- **破坏性操作模态阻断**（迁移/改密/移除/销毁前），靠各自原子协议保证"绝无半态"。
- 密码输入由用户在本机原生输入框完成，助手不代填。

## 7. 找回 —— 本地无后门（用户拍板：忘密=彻底没了）

- **铁律（你已确认）**：本地**不留任何"光靠密码就能解"的后门**。后果：**忘密 / 清 TPM / 重装系统 / 换主板或 CPU(fTPM 绑 CPU) / TPM 固件升级失效 = 本地数据彻底找不回**（完整枚举，红队补全）。
- **撤掉 v3 的错误建议**：不再把"导出未加密备份"设为推荐/默认动作——那份明文备份正好喂饱 §1 要防的设备级对手（捡到电脑直接打开备份），且与"无后门"自相矛盾。
- **本期的防手滑（不是后门，是误删兜底）**：
  - **开销毁前强制先备份一次** + **销毁默认走回收站隔离区**（§5）。
  - 备份**新鲜度感知**：记录上次备份时间，设置页常驻显示、超期软提醒；高危操作前校验近期备份（红队：一次性备份抵不住数月后换机/忘密）。
  - 用户若要导出备份：明确这是**关闭加密的等价物**、明标"不受密码保护、请移到离线介质且别和电脑一起放"，并在状态行常驻提示"存在 1 份未加密备份于 <路径>"。**默认不主动生成。**
- **可选恢复码**：默认**不做**（本质是离线可用的后门，与"真硬"冲突，你已选真硬）。列开放项，若将来要再议。
- **真正的找回靠未来端到端云同步**（§8）——但注意其单机限制。

## 8. 未来端到端云同步（方案三，本期后置，留接口）

- **目标（像 WhatsApp）**：数据在你设备上加密、服务器只存乱码、永远看不到明文；同一把 DEK。
- **跨设备密钥分发 = 每设备密钥包裹 + 新设备走已有端授权**（v2 红队定，避免把 E2E 安全坍缩到密码强度、可 per-device 撤销）。
- **⚠️ 单设备用户的死锁（v3 红队，必须摊开）**：目标用户多为**单台电脑**。每设备包裹意味着云端那份只被这台设备的密钥包裹；**这台坏了/清了芯片，没有"另一台已存活设备"来授权新设备 → 云端 blob 也永远解不开**。即"云找回"在最该兜底的场景(唯一设备损坏)失效。
- **取舍（将来拍板）**：要让单设备用户也能云找回，必须引入一把**不绑设备的恢复因子**（密码派生 KEK 或恢复码包裹 DEK 存云端）——但这又把强度降回密码爆破。**"对别人很难恢复" vs "单设备用户对自己能找回" 不可兼得**，留给云同步阶段定。本期诚实告知：现在没有任何云兜底。
- 与本期衔接：DEK 已是"随机、芯片绑定"的独立钥匙，将来加云时向云端做"每设备包裹"即可，不必重构本地。

## 9. 已有明文库的加密迁移 —— Rust 内原子协议（红队：别经 JS 插件）

现存明文 `heng.db`（WAL 还有 `-wal/-shm`）。**整个迁移在 Rust 桥内完成，不引 tauri-plugin-fs**（它拿不到想要的原子 replace 语义；Rust `std::fs::rename` 同卷即可原子替换已存在文件，或 `ReplaceFileW`）。
1. 新文件写临时名 + `fsync`。
2. 校验密文能解、与明文逐行一致。
3. **原子 rename 顶替** + fsync 目录。
4. 仅当全成功，**先把 `heng.db` 原子改名 `.bak`、确认无误再 unlink**（不原地覆盖）；**一并清 `-wal/-shm` 明文页**。
5. **启动自愈**：发现 `.tmp` 残留 / 元数据缺失 / 校验失败而明文仍在 → 判未完成、回退明文重试，绝不进"只剩半截密文"死局。
- **SSD 残留诚实声明**：安全删除在 SSD 上不保证物理擦除，best-effort；在意者建议整盘加密/擦盘。
- 移除密码(密文→明文)走反向同协议。

## 12. 实施分期（先 Spike，可证伪子项）

1. **Spike（先做，最可能翻车的先验）**：
   - **构建链**：干净 Windows MSVC + CI runner 上 `rusqlite`(bundled-sqlcipher-vendored-openssl) `cargo build` 通过、产物能开 SQLCipher 库（OpenSSL/Perl 工具链最可能 fail）。
   - **芯片封装**：NCrypt PCP 建非导出 key + 密码(PIN)授权 + 加密会话 + `NCryptDeleteKey` 删钥匙跑通。
   - **硬路**：tss-esapi/TBS 建 NV 单调计数器 + PolicyNV("计数<N")绑封装对象、用户态可增计数——**能否做出来是硬清空成败的关键**。
   - **原子替换**：Rust `std::fs::rename` 在目标卷对已存在文件原子替换 + 目录 fsync。
2. **crypto/芯片模块 + 单测**：封装/解封、三类失败分流、销毁计数与 sentinel、迁移一致性、改密码 TPM 对象切换原子。
3. **接 store（SQLCipher 桥）+ 解锁屏（三态/三类失败/销毁终态）+ 设置 + 备份导出**：每步实测桌面持久化。
4. **迁移既有明文库**（§9）+ 文档（讲清威胁模型全部边界、"忘密=彻底没了"、强制备份）。
5. **验收**：改密码/移除/迁移**任意时点 kill-9 仍可用旧或新态**（分别覆盖三条路径）；输错 N 次确实销毁(硬路：拔电也躲不过)；坏块/芯片不可用**不**误触销毁；磁盘扫描无意外明文窗口。
6. **端到端云同步另排**，复用 DEK + §8 每设备包裹边界（含单设备恢复因子取舍）。
